your task will be to tidy up the evaluation suite so that the entire process (install dependencies for each benchmark, fetch data (either small or medium) for each benchmark, and run each benchmark) can be run end to end with a convenient script that also supports different customization modes. also, the current infrastructure is not very robust (i.e. when incr is killed in the middle of execution, the instrumentation script needs to restore the original file). your task will also be to make it more robust.

specifically, do the following:
1. first run incr manually on a few simple commands so you can understand how it works and see what artifacts it generates.
2. for each benchmark, inspect it and understand how it works. then, write a script to install the dependencies and also min and normal (normal is the current size used for evaluation). some of the benchmarks do not have min inputs yet, for those generate very simple min inputs, possibly by taking a small chunk from the beginning of the small inputs. make all of these scripts robust to cleanup and leftover artifacts.
3. for each benchmark, write a script that will automatically install dependencies and download data, and have another script that will execute the benchmark (make it configurable so you can execute bash mode, incr mode, or both). make these new scripts on top of the clean and execute just so we can preserve the old code. these new scripts should be especially robust to errors and cases such as being killed. for example, incr generates a lot of artifacts when running, so they should make sure to clean up artifacts. also, the insert script instruments a bash script and replaces it while running, so when a program is killed it should make sure everything is cleaned up and restored.
4. after you have all of these scripts, verify that they all work with the minimum input sizes on each benchmark (good for confirming accuracy without taking too long)
5. then, generate a global script that will install dependencies / download data for each benchmark using subscripts if they were not already installed / downloaded, and then run all the benchmarks using either min or small inputs. add an option to run the benchmarks excluding the ones that require annoying setup such as the one that requires the image segmentation model and the one that requires the openai api key, and have another mode that also runs these. the goal is to make it simple for artifact evaluators to call this script and then run the majority of the benchmarks that dont require complex setup and are less error prone.

some key things to keep in mind are
1. you are dealing with a potentially buggy bash environment here, so things will hang and fail. when you are running things, run them in the background and monitor them frequently to see if they are hanging.
2. the run scripts may take a very long time because the data sizes used for evaluation are very large. test with small inputs first to validate before moving on to the full large inputs.
3. some of the benchmarks require external dependencies or complex installs. be mindful of this, especially when designing the convenient run scripts. also, one of the benchmarks requires an openai API key. this is why the script should have two modes, one to run the benchmarks that are easy to run and one that runs all the benchmarks including the annoying ones (specifically the one requiring segment anything model and the openai image annotation).

make a plan to do this. also, after you have written the plan, copy the plan into the agents directory. use the agents directory as scratch space to store context if you need it while working. specifically, after completing a major step, update the agents directory with context so that if this session is killed another agent can pick it up. commit your work and push after every major improvement also.

I cannot emphasize enough that a lot of junk and artifacts will be generated while you are working, whether that is in incr cache directories or in the /tmp directory. stuff will also hang, make sure you are on top of detecting this and cleaning up artifacts, which may potentially require sudo (if it is an overlayfs directory).
